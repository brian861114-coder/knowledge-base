Chapter 1 · 基礎

HTML 結構層 HTML

186 行 · 定義頁面「有什麼元素」—— 如同房屋的鋼筋水泥骨架。本章會從零開始，用實際例子帶你認識每一個關鍵標籤。

1.1 什麼是 HTML？用一個比喻開始

想像你要蓋一間房子。在還沒貼磁磚、刷油漆、拉電線之前，你必須先決定：這裡有一面牆、那裡有一扇窗、轉角要放一扇門。HTML 就是這個「決定結構」的步驟。

🏗 一句話總結：HTML 用標籤（Tag）描述頁面上的元素，只定義「有什麼」，不處理「長怎樣」（那是 CSS）或「怎麼動」（那是 JS）。

1.2 HTML 檔案的基本骨架

每一個 HTML 檔案都從這個基本架構開始。你可以把它想像成一份文件的「標準表頭」：

<!DOCTYPE html>                          <!-- 告訴瀏覽器：這是 HTML5 文件 -->
<html lang="zh-Hant">                    <!-- 整份文件的根，lang 指定語言 -->
<head>
  <meta charset="UTF-8">                 <!-- 編碼：讓中文正常顯示 -->
  <meta name="viewport" content="...">   <!-- 手機版縮放設定 -->
  <title>頁面標題</title>                <!-- 瀏覽器分頁上的文字 -->
  <link rel="stylesheet" href="...">     <!-- 載入 CSS 樣式檔 -->
</head>
<body>
  <!-- 這裡放所有「使用者看得到」的內容 -->
  <script src="..."></script>           <!-- 載入 JavaScript -->
</body>
</html>
💡 關鍵觀念：<head> 放的是「給瀏覽器的設定」（不可見），<body> 放的才是「使用者看到的內容」。<script> 放在 <body> 最後面，是為了確保 HTML 都載入完了才執行 JS。

1.3 認識標籤：點擊下方按鈕查看說明

HTML 的每一個 <標籤名> 都代表一種元素類型。本專案中最常用的標籤如下：

<div> <button> <input> <svg> <aside> <header> <main> <section> <head> <script>
<div> — 萬用容器（division 的縮寫）
最常用的標籤，本身沒有任何視覺效果，純粹用來「把一群元素裝在一起」。本專案中 app-shell、left-rail、content-shell 都是 div。

1.4 標籤的兩種形式

形式語法例子
成對標籤（container tag）<tag>內容</tag><h1>標題文字</h1>、<div>...</div>
自閉合標籤（self-closing tag）<tag> 或 <tag /><img src="...">、<input type="text">、<br>

判斷方式：如果標籤中間需要「包住內容」，就用成對標籤。如果只是「插入一個東西」，就用自閉合標籤。

這兩個 屬性（attribute） 是前端開發中最重要的概念之一。

📌 id — 獨一無二的身份證字號

一個頁面中，同一個 id 只能出現一次。CSS 用 #id名 選取，JS 用 getElementById("id名") 找到它。

1.7 常見錯誤與注意事項

⚠ 初學者最常犯的錯：
1. 忘記關閉標籤 — <div>...</div> 缺了結尾的 </div>
2. id 重複使用 — 同一個 id 在頁面中出現兩次，JS 只會找到第一個
3. 巢狀錯誤 — <p><div>...</div></p> 是不合法的（p 裡面不能放 div）
4. <script> 放錯位置 — 放在 <head> 裡又沒加 defer，會導致 JS 在 HTML 載完前執行而出錯

CSS 樣式層 CSS

999 行 · 控制元素「長什麼樣子」—— 如同室內設計的裝潢、配色與家具擺設

2.1 CSS 解決什麼問題？

在 CSS 出現之前，網頁的樣式是直接寫在 HTML 標籤裡的（例如 <font color="red">）。當網站有 100 個頁面、每個頁面有 50 個紅色標題時，想改成藍色就得改 5,000 個地方 😱

CSS 的核心價值：把「樣式規則」集中管理在一個檔案裡，改一個地方，全站一起變。

2.2 CSS 變數（設計 Token）— 本專案最聰明的設計

傳統寫法：每個地方直接寫顏色代碼 color: #007AFF;。如果之後要換顏色，得搜尋整個檔案一個一個改。

本專案的寫法：先把所有顏色定義成變數，之後全部用 var(--變數名) 引用。

2.7 CSS 的優先權（Specificity）— 為什麼有些樣式會覆蓋不掉？

當多個 CSS 規則同時指向同一個元素時，瀏覽器會按照「權重」決定誰贏：

#id（權重 100） > .class（權重 10） > tag（權重 1）

例如：#graphSvg .graph-node 的權重是 100+10=110，會覆蓋掉單純的 .graph-node（權重 10）

Chapter 3 · 基礎

JavaScript 行為層 JavaScript

1,720 行 · 讓頁面「活起來」—— 如同水電管線與開關系統。這是三個語言中唯一能「做決定」和「記住狀態」的。

3.1 為什麼需要 JavaScript？HTML+CSS 的極限在哪？

HTML 定義結構，CSS 定義外觀——但兩者都是靜態的。一旦頁面載入完成，它們就「定型」了。

🧠 JavaScript 能做而 HTML+CSS 做不到的事：
1. 回應使用者操作（點擊、拖曳、鍵盤輸入、滾輪）
2. 記住狀態（目前是哪個模式？縮放到多少？選了哪個節點？）
3. 動態改變畫面（新增/刪除/修改 HTML 元素、改變 CSS 樣式）
4. 載入外部資料（從 JSON 檔讀取圖譜資料）
5. 執行數學計算（力導向佈局演算法、座標轉換）

3.2 核心模式：State → Render（狀態 → 渲染）

這是現代前端開發中最重要的設計模式。它的邏輯非常簡單：

📦 把所有「會變的東西」集中在 state 物件裡 → 🔄 任何改變都透過 render() 重繪整個畫面

這樣做的好處：你永遠不用擔心「這個東西改了但那個東西忘了更新」——每次 render() 都會從 state 重新計算整個畫面。

👇 親自體驗：點擊按鈕改變 state，觀察輸出變化

總覽模式 焦點模式 🔍 +0.2 🔎 -0.2 選取節點
{ "mode": "overview", "zoom": 1.0, "selectedNode": null }

每次點擊 → state 物件即時更新 → 在真正的 app.js 中會接著呼叫 render() 重繪整個 SVG 圖譜

③ 資料載入 — fetch() 與 async/await

這是最關鍵的非同步操作。JavaScript 不會「卡住等資料回來」，而是用 await 告訴它「等這個做完再繼續」：

async function loadData() {
  // fetch() 回傳一個 Promise（承諾之後會給你結果）
  const graphResp = await fetch("../physics_graph.json");
  const graphData = await graphResp.json();

  const detailResp = await fetch("../physics_note_details.json");
  const detailData = await detailResp.json();

  // 建立快速查表：節點 ID → 節點資料（含 x, y 座標）
  for (const node of graphData.nodes) {
    state.nodeMap.set(node.id, node);
  }
  render();  // 資料準備好了，開始畫！
}
⏳ async/await 白話解釋：想像你去咖啡店點餐。「async function」就是你走進店裡，「await fetch()」就是你跟店員點餐然後在旁邊等，「await response.json()」就是店員把咖啡做好遞給你。你沒有卡在櫃台不動，只是「等咖啡的期間可以先滑手機」（JS 可以處理其他事）。

④ 圖譜渲染（~400-700 行）

render() 總指揮renderNodes() 畫圓renderEdges() 畫線renderRings() 領域圈renderMiniMap() 小地圖relaxLayout() 排版

render() 每次被呼叫時，會按照順序執行以上所有子函數。你可以把它理解成「動畫的每一幀」——只要 state 有變化，就重新畫一幀。

⑤ 互動處理（~800-860 行）

// 拖曳平移的核心邏輯
function onPointerDown(event) {
  state.dragging.active = true;           // 開始拖曳
  state.dragging.startX = pointer.x;      // 記錄起始位置
  state.dragging.basePanX = state.panX;   // 記錄起始平移量
}
function onPointerMove(event) {
  if (!state.dragging.active) return;   // 沒在拖曳就跳過
  const deltaX = pointer.x - state.dragging.startX;
  state.panX = state.dragging.basePanX + deltaX;  // 平移 = 基準 + 偏移
  updateViewportTransform();             // 立即更新畫面
}

三者如何協作？

追蹤兩個完整場景，看看 HTML、CSS、JS 如何分工合作

場景一：使用者點擊圖譜節點

1HTML 提供元素：<circle class="graph-node" data-node-id="newton_2"> 已經在 SVG 中

2CSS 提供樣式：.graph-node { cursor: pointer; } 讓滑鼠移到節點上變成手指

3JS 處理點擊：addEventListener("click") → 更新 state.selectedNodeId → 呼叫 render()

4JS 更新畫面：render() → 被選中節點加上 .is-selected class → CSS .graph-node.is-selected { stroke: var(--navy); stroke-width: 3px; } 生效 → 節點出現藍色外框

場景二：使用者滾輪縮放圖譜

1JS 監聽滾輪：addEventListener("wheel", onWheel)

2JS 計算新縮放值：根據滾輪方向和速度計算新的 state.zoom

3JS 更新 SVG transform：viewport.setAttribute("transform", "translate(...) scale(1.5)")

4CSS 無動作：這個場景中 CSS 不需要做任何事——SVG 的 transform 是直接寫在元素屬性上的

場景三：使用者在搜尋框打字

1HTML 提供搜尋框：<input id="searchInput" type="search">

2CSS 提供樣式：.searchbox input { border-radius: 12px; ... }

3JS 監聽輸入：addEventListener("input", ...) → 過濾節點 → renderSearchResults()

4JS 動態產生 HTML：用 document.createElement() 建立搜尋結果下拉列表，插入到 DOM 中

💡 核心心法：HTML 定義「初始狀態有什麼」→ CSS 定義「任何狀態下長怎樣」→ JS 定義「狀態如何改變」以及「改變後如何更新 HTML/CSS」。三者各司其職，透過 DOM 串聯。

nodeMap：用空間換時間

一條邊記錄的是 {source: "A", target: "B"}——只存了ID，沒有座標。但畫線需要座標。如果每次都要從 500 個節點中搜尋「ID 是 A 的節點在哪」，每次 render 都要搜幾百次，會很慢。

解法：在載入資料時，把所有節點放進一個 Map，用 ID 當 key，節點資料當 value。之後 nodeMap.get("A") 瞬間就能拿到 {x, y}。

// 建立 nodeMap：O(1) 查詢，而非 O(n) 搜尋
const nodeMap = new Map();
for (const node of data.nodes) {
  nodeMap.set(node.id, node);
}
// 之後任何地方需要節點座標：
const nodeA = nodeMap.get(edge.source);  // 瞬間！

edgeIndex：加速關係查詢

類似 nodeMap 的概念，但用於邊。當使用者選中一個節點進入焦點模式時，需要找出所有跟這個節點相關的邊。用 edgeIndex 可以快速查到，而不需要遍歷所有邊。

資料流完整生命週期（五階段）

1. 載入（Load） — fetch() 讀取硬碟上的 JSON 文字 → response.json() 轉成 JS 物件

2. 索引（Index） — 建立 nodeMap（ID→節點）和 edgeIndex（分類邊）

3. 過濾（Filter） — 根據目前模式（總覽 all / 焦點 subset）篩選要顯示的節點和邊

4. 佈局（Layout） — relaxLayout() 用力導向演算法微調節點位置，避免重疊

5. 渲染（Render） — 把 JS 物件轉成 SVG <circle> 和 <line>，寫入 DOM

核心演算法（逐行解說）

function renderMarkdown(text) {
  const lines = text.split('\n');        // ① 把文字切成一行一行
  const blocks = [];                    // ② 準備一個空陣列放結果
  let i = 0;                           // ③ 從第 0 行開始處理

  while (i < lines.length) {           // ④ 一行一行處理，直到最後一行
    const line = lines[i];

    // 檢查是否為標題（## 開頭）
    const h = line.match(/^(#{1,6})\s+(.+)$/);
    if (h) {
      const lv = h[1].length + 1;       // ## → h3, ### → h4
      blocks.push(`<h${lv}>${h[2]}</h${lv}>`);
      i++; continue;
    }

    // 檢查是否為無序列表（- 開頭）
    if (/^- /.test(line)) {
      const items = [];
      while (i < lines.length && /^- /.test(lines[i])) {
        items.push(`<li>${lines[i].slice(2)}</li>`);
        i++;                           // 連續的 - 行合併成一個 <ul>
      }
      blocks.push(`<ul>${items.join('')}</ul>`);
      continue;
    }

    // 以上都不符合 → 當作段落處理
    if (line.trim()) {
      blocks.push(`<p>${line}</p>`);
    }
    i++;
  }
  return blocks.join('');              // ⑤ 把所有區塊串成一個字串回傳
}

MathJax 的 Token 保護機制

數學公式（$...$）包含特殊字元，如果在 Markdown 轉換過程中被當成一般文字處理，會被破壞。解法是「先藏起來，轉換完再放回去」：

① 掃描 → 找出所有 $...$，替換成 __TOKEN_0__、__TOKEN_1__…

② 轉換 → 對剩餘文字執行 Markdown → HTML 轉換（token 不會被誤判為標題或列表）

③ 還原 → 把 __TOKEN_0__ 換回原始的 $F=ma$

④ 渲染 → MathJax 自動掃描頁面，把 $...$ 渲染成真正的數學符號
